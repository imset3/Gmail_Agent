Gmail Multi-Agent Email Assistant

실제 Gmail API 또는 더미 메일 데이터를 사용해 메일을 병렬 분류하고, 자동답장·스팸 등록·일정 충돌·사용자 판단 필요 항목을 처리하는 과제 제출 문서입니다.

개요 핵심 개념 실행 방법 구조 처리 흐름 Windows Gmail API 산출물 오류 해결 GitHub 제출 제작 프롬프트

1. 프로젝트 개요

이 프로젝트는 메일함을 자동으로 분석하는 멀티 Agent 시스템입니다. 입력은 더미 JSON 메일 또는 실제 Gmail API이며, 출력은 콘솔 리포트와 JSON/Markdown 리포트입니다.

핵심 목표는 사람이 직접 메일함을 열어 하나씩 판단하던 작업을 Agent 단위로 나누어 자동화하는 것입니다. 단순히 문자열을 분류하는 프로그램이 아니라, 메일 수집, 의미 판단, 일정 충돌 확인, 답장 생성, 스팸 등록, 리포트 작성이라는 여러 업무를 작은 Agent들이 협력해 처리하도록 설계했습니다.

2. 핵심 개념

멀티 Agent란?

멀티 Agent 구조는 하나의 거대한 함수가 모든 일을 처리하는 방식이 아니라, 역할이 분리된 여러 Agent가 순서대로 또는 병렬로 협력하는 구조입니다. 이 프로젝트에서는 메일 가져오기, 분류, 스팸 처리, 일정 확인, 답장 생성, 리포트 생성이 각각 별도 Agent의 책임입니다.

일반 프로그램 방식이 프로젝트의 Agent 방식
하나의 코드 흐름에서 모든 조건문을 처리Agent별 역할을 나누고 결과를 조합
기능 추가 시 기존 코드가 복잡해짐새 Agent 또는 Provider를 추가해 확장 가능
실제 Gmail과 더미 데이터가 섞이기 쉬움입력 Provider만 바꾸고 나머지 처리 로직은 재사용

LLM과 규칙 기반 fallback

메일 의미 판단은 OpenAI 또는 Ollama 같은 LLM Provider가 담당할 수 있도록 만들었습니다. 다만 API 키가 없거나 로컬 모델이 준비되지 않은 환경에서도 과제 시연이 가능해야 하므로, 규칙 기반 fallback을 함께 넣었습니다.

  • LLM 사용 가능: 제목과 본문 의미를 보고 스팸, 무응답, 의사결정 필요, 자동답장 등을 판단합니다.
  • LLM 사용 불가: 광고 키워드, 일정 표현, 승인/검토 문구, 친근한 답장 요청 문구를 기준으로 안정적으로 분류합니다.
  • 장점: 외부 API 설정이 없어도 결과가 재현되며, 설정이 있으면 더 자연스러운 판단으로 확장됩니다.

더미 모드와 Gmail 모드를 분리한 이유

과제 평가는 실행 재현성이 중요합니다. 실제 Gmail은 계정, OAuth, 메일함 상태에 따라 결과가 달라질 수 있으므로, 기본 시연은 더미 데이터로 동일한 결과가 나오게 했습니다. 반면 Gmail API 코드는 실제 연동 능력을 보여주기 위해 같은 파이프라인에 연결했습니다. 특히 review-gmail 모드는 실제 메일을 읽어 검토만 하고 Draft나 스팸 DB 변경은 하지 않으므로 안전한 실제 계정 시연에 적합합니다.

Agent 역할

  • MailFetchAgent: 더미 JSON 또는 Gmail API에서 메일을 가져옵니다.
  • ClassificationAgent: LLM 또는 규칙 기반 fallback으로 메일을 분류합니다.
  • SpamAgent: 스팸 발신자를 DB에 저장합니다.
  • ScheduleAgent: 메일 속 날짜와 기존 일정을 비교해 충돌을 찾습니다.
  • ReplyAgent: 자동답장 결과를 더미 발신함 또는 Gmail Draft로 저장합니다.
  • ReportAgent: 콘솔, JSON, Markdown 리포트를 생성합니다.

5. 실행 로직과 처리 흐름

실행은 start.py 또는 main.py에서 시작됩니다. 사용자가 더미 모드, Gmail 검토 전용 모드, Gmail 실행 모드 중 하나를 선택하면 MailFetchAgent가 입력 데이터를 가져오고, 이후에는 입력 출처와 상관없이 같은 Agent 파이프라인을 통과합니다.

  1. 메일을 가져옵니다. 더미 모드는 dummy_mails/emails.json, Gmail 모드는 Gmail API를 사용합니다.
  2. asyncio.gather로 메일별 분류 작업을 병렬 실행합니다.
  3. 스팸은 spam_db.json에 등록합니다.
  4. 의사결정 필요 메일은 일정 충돌과 사용자 판단 상태를 기록합니다.
  5. 자동답장 메일은 더미 모드에서 sent_mails.json에 저장하고, Gmail 실행 모드에서는 Draft 초안을 생성합니다.
  6. Gmail 검토 전용 모드에서는 Draft 생성과 스팸 DB 저장을 건너뛰고 검토 결과만 남깁니다.
  7. 최종 결과를 콘솔과 output/results.json, report/email_report.md에 저장합니다.

병렬 처리 원리

메일 7개가 있을 때 하나씩 순서대로 LLM/규칙 판단을 기다리면 느릴 수 있습니다. 그래서 분류 단계는 asyncio.gather로 메일별 판단 작업을 동시에 예약합니다. 이후 스팸 DB 저장, 더미 답장 저장, 리포트 작성처럼 파일을 갱신하는 단계는 결과를 모은 뒤 순서대로 처리해 데이터가 꼬이지 않게 했습니다.

사용자 판단이 필요한 이유

모든 메일을 자동 처리하면 위험합니다. 예산안 선택, 회의 일정 충돌, 승인 요청처럼 사용자의 의사결정이 필요한 메일은 자동 답장하지 않고 decision_required로 분리합니다. 이 방식은 자동화와 안전성을 함께 만족시키기 위한 설계입니다.

7. Gmail API 적용 방법과 원리

  1. Google Cloud Console에서 Gmail API를 활성화합니다.
  2. OAuth Client를 Desktop app으로 생성합니다.
  3. 다운로드한 JSON 파일명을 credentials.json으로 바꿉니다.
  4. config/credentials.json 위치에 넣습니다.
  5. python3 start.py --auth 또는 AUTH_GMAIL_WINDOWS.bat을 실행해 인증합니다.

Gmail API는 두 단계로 나누어 사용할 수 있습니다. 검토 전용 모드는 실제 Gmail 메일을 읽고 분류/리포트만 생성합니다. Gmail 실행 모드는 자동답장 가능한 메일에 대해 Gmail Draft 초안까지 생성합니다. 두 모드 모두 실제 발송은 하지 않습니다.

검토 전용 실행
python3 start.py --review-gmail --limit 20
python3 main.py review-gmail --limit 20 --interactive --report

Draft 생성까지 허용하는 Gmail 실행
python3 start.py --gmail --limit 20
python3 main.py run --source gmail --limit 20 --interactive --report

OAuth 인증 원리

Gmail API는 사용자의 메일함에 접근하므로 비밀번호를 코드에 넣지 않습니다. 대신 Google Cloud에서 발급받은 credentials.json으로 OAuth 인증 창을 열고, 사용자가 브라우저에서 권한을 승인하면 token.json이 생성됩니다.

  • credentials.json: 이 앱이 어떤 Google OAuth 앱인지 알려주는 설정 파일입니다.
  • token.json: 사용자가 승인한 뒤 생성되는 접근 토큰/갱신 토큰 파일입니다.
  • 두 파일은 개인정보이므로 GitHub에 올리지 않도록 .gitignore에 제외했습니다.

메일 읽기 원리

Gmail 모드에서 MailFetchAgent는 Gmail API의 message list/get 흐름을 사용합니다. 먼저 inbox 메시지 ID 목록을 가져오고, 각 ID에 대해 제목, 발신자, 수신자, 날짜, 본문을 읽어 더미 메일과 같은 형태의 EmailItem 구조로 변환합니다. 이렇게 하면 더미 모드와 Gmail 모드가 같은 분류 Agent를 공유할 수 있습니다.

Gmail API 원리
1. users.messages.list(userId="me", q="in:inbox")
2. users.messages.get(userId="me", id=message_id, format="full")
3. Gmail header/body를 EmailItem JSON 구조로 변환
4. ClassificationAgent -> SpamAgent/ScheduleAgent/ReplyAgent로 전달

실제 Gmail 검토 전용 모드 원리

review-gmail 명령은 Gmail API에서 실제 inbox 메일을 가져온 뒤 같은 분류 Agent를 실행하지만, 후처리 단계에서 외부 변경 작업을 막습니다. 즉, 자동답장 후보가 있어도 Draft를 만들지 않고 review_only_no_draft 상태로 기록하며, 스팸 후보도 로컬 스팸 DB에 등록하지 않습니다.

Draft 생성 원리

자동답장 가능한 메일이 Gmail 모드에서 발견되면 실제 발송하지 않고 Draft만 만듭니다. 답장 본문을 MIME 이메일로 만들고 Base64 URL-safe 형식으로 인코딩한 뒤 Gmail Draft API에 전달합니다. 이러면 Gmail의 임시보관함에 초안이 생기고, 사용자가 직접 확인 후 보낼 수 있습니다.

Gmail Draft 생성 흐름
1. EmailMessage에 To, From, Subject, Body 설정
2. MIME 메시지를 base64.urlsafe_b64encode로 인코딩
3. users.drafts.create(userId="me", body={message: {raw, threadId}})
4. 결과 draft id를 output/reply_drafts.json에 기록

9. 발생 가능 오류와 해결

오류원인해결
Python이 설치되어 있지 않습니다 Windows PATH에 Python이 없습니다. Python 3.10 이상 설치 후 Add python.exe to PATH를 체크합니다.
ModuleNotFoundError: google... Gmail API 패키지가 설치되지 않았습니다. pip install -r requirements.txt 또는 RUN_GMAIL_WINDOWS.bat 실행.
config/credentials.json 파일이 필요합니다 Gmail OAuth JSON 파일이 없습니다. 다운로드한 OAuth JSON을 config/credentials.json으로 저장합니다.
403 access_denied OAuth 앱이 Testing 상태인데 테스트 사용자에 Gmail 계정이 없습니다. Google Cloud Console의 OAuth consent screen에서 Test users에 로그인 계정을 추가합니다.
Gmail은 되는데 답장이 발송되지 않음 의도된 안전 정책입니다. 실제 발송 대신 Gmail Draft 초안만 생성합니다. Gmail에서 직접 확인 후 발송하면 됩니다.
실제 Gmail 검토만 하고 싶은데 Draft가 생길까 걱정됨 일반 Gmail 실행 모드는 Draft 생성이 가능합니다. python3 start.py --review-gmail --limit 20 또는 REVIEW_GMAIL_WINDOWS.bat을 사용하면 Draft를 만들지 않습니다.
LLM이 실행되지 않음 OpenAI/Ollama 환경변수가 없습니다. 기본 규칙 fallback으로 정상 실행됩니다. LLM을 쓰려면 OPENAI_API_KEY 또는 OLLAMA_MODEL을 설정합니다.
GitHub에 인증 파일이 올라갈까 걱정됨 OAuth 파일은 민감 정보입니다. .gitignore에 config/credentials.json, config/token.json을 제외 처리했습니다.

목표:
- 실제 Gmail API 또는 더미 JSON 메일 데이터를 사용한다.
- 여러 Agent가 메일을 병렬로 처리한다.
- 메일을 스팸, 무응답, 의사결정 필요, 자동답장 가능, 일정 업데이트로 분류한다.
- 캡쳐 예시처럼 터미널에 처리 진행상황과 최종 리포트를 보기 좋게 출력한다.
- 기본 시연은 Gmail 인증 없이 더미 데이터만으로 재현 가능해야 한다.
- Gmail 모드에서는 실제 메일 발송을 하지 말고 Draft 초안만 생성한다.
- 실제 Gmail 검토 전용 모드를 별도로 제공한다. 이 모드는 Gmail API로 내 inbox 메일을 불러와 분류/리포트만 만들고 Draft 생성이나 스팸 DB 변경은 하지 않는다.

필수 폴더/파일:
- main.py: 상세 CLI 진입점
- start.py: 사용자가 쉽게 실행할 수 있는 메뉴형 진입점
- agent.py: Agent, Gmail Client, LLM Provider, 리포트 생성 로직
- dummy_mails/emails.json: 수신 더미 메일
- dummy_mails/sent_mails.json: 더미 모드 자동답장 발신함
- data/schedule.json: 일정 충돌 판단용 일정 데이터
- data/spam_db.json: 스팸 발신자 DB
- output/results.json: 전체 처리 결과
- output/reply_drafts.json: 답장 초안 결과
- report/email_report.md: 제출용 Markdown 리포트
- docs/submission_guide.html: 개념, 실행 방법, 오류 해결, API 원리를 설명하는 제출용 HTML
- requirements.txt, README.md, .gitignore
- Windows용 RUN_WINDOWS.bat, RUN_DUMMY_WINDOWS.bat, AUTH_GMAIL_WINDOWS.bat, REVIEW_GMAIL_WINDOWS.bat, RUN_GMAIL_WINDOWS.bat

Agent 구성:
- MailFetchAgent: dummy 또는 Gmail에서 메일을 가져온다.
- ClassificationAgent: OpenAI 또는 Ollama LLM으로 판단하되, API가 없으면 규칙 기반 fallback으로 분류한다.
- SpamAgent: 스팸 메일 발신자를 data/spam_db.json에 저장한다.
- ScheduleAgent: 메일 본문 날짜/시간과 data/schedule.json을 비교해 일정 충돌을 찾는다.
- ReplyAgent: 자동답장 가능한 메일에 대해 답장 본문을 생성한다. 더미 모드에서는 sent_mails.json에 저장하고, Gmail 모드에서는 Draft만 생성한다.
- ReportAgent: 콘솔 출력, JSON 결과, Markdown 리포트를 생성한다.

데이터 구조:
- dummy_mails/emails.json은 emails 배열을 사용한다.
- 각 메일 필드는 sender_name, sender_email, subject, body, to, cc, date, is_read, thread_id, msg_id를 포함한다.
- data/schedule.json은 events 배열을 사용한다.
- 각 일정 필드는 from, to, event_name, event_detail을 포함한다.
- sent_mails.json은 sent_mails 배열을 사용하고, to, from, subject, body, in_reply_to_msg_id, thread_id, date, status를 저장한다.

실행 요구:
- python3 start.py 실행 시 메뉴가 떠야 한다.
- python3 start.py --dummy 로 더미 모드가 바로 실행되어야 한다.
- python3 start.py --review-gmail --limit 20 으로 실제 Gmail 검토 전용 모드가 실행되어야 한다.
- python3 start.py --gmail --limit 20 으로 Gmail 모드가 실행되어야 한다.
- python3 start.py --auth 로 Gmail OAuth 인증을 수행해야 한다.
- Windows에서는 RUN_WINDOWS.bat 더블클릭으로 실행 가능해야 한다.

Gmail API 원리:
- config/credentials.json을 사용해 OAuth 인증을 수행한다.
- 인증 후 config/token.json을 생성한다.
- Gmail readonly와 compose 권한만 사용한다.
- users.messages.list/get으로 메일을 읽고, users.drafts.create로 답장 초안만 만든다.
- review-gmail 모드에서는 users.messages.list/get으로 메일을 읽고 분류하되 users.drafts.create를 호출하지 않는다.
- credentials.json과 token.json은 .gitignore에 넣어 GitHub에 업로드하지 않는다.

출력 요구:
- 처리 중 [1/7] 🚨 스팸 | 제목 형식으로 진행상황을 출력한다.
- 최종 리포트에는 총 처리, 의사결정, 일정추가, 답장, 무응답, 스팸 카운트를 보여준다.
- 의사결정 필요 메일은 발신자, 제목, 일정 충돌, 사용자 판단 대기 상태를 보여준다.
- 자동답장 메일은 답장 본문과 저장 위치를 보여준다.

테스트:
- 더미 메일 7개를 만들어 스팸 2개, 무응답 2개, 자동답장 1개, 의사결정 필요 2개가 나오게 한다.
- unittest로 분류 카운트, 자동답장 감지, 일정 충돌 감지를 검증한다.

문서:

- docs/submission_guide.html에는 프로젝트 개념, 멀티 Agent 구조, 실행 로직, Gmail API 원리, 오류 해결, GitHub ZIP 제출 방법, 이 프롬프트를 포함한다.
